1 · Le paysage consolidé

Une courbe par modèle, un point par niveau d'effort (low → max). Axe X = coût relatif, axe Y = qualité relative — les deux calculés à l'identique : on ne compare jamais deux scores bruts de benchmarks différents (échelles incompatibles), on prend le ratio de score entre deux couples au sein d'un même benchmark, on chaîne sur le graphe de liaison (§3) et on consolide, ancré à Opus 4.8 @medium = 1,0. L'ellipse autour de chaque point = ±1σ mesuré (largeur = σ coût, hauteur = σ qualité). Ellipse plate = valeur vue par une seule source. Haiku 4.5 n'est pas tracé ici (voir §2, §6 et le Pareto).

Les deux axes viennent de la même méthode couple-atomique (comme le coût) : ratios de score à tâche identique → graphe de liaison → consolidation. La montée effort→qualité est donc mesurée (CursorBench, OSWorld, Braintrust, zenn… balaient l'effort), pas un a priori ; les courbes sont monotones. Bornes et quadrillage recalculés dynamiquement.

2 · La matrice normalisée

Coût relatif par modèle × effort (base Opus 4.8 @medium = 1,00), trié par intelligence décroissante. Ces valeurs sont calculées (pas saisies à la main) à partir des ratios mesurés à tâche identique : chaque cellule est résolue couple-atomiquement le long du graphe de liaison (§3). Intensité = coût ; le second nombre est l'IC = ±1σ (écart-type des estimations mono-source qui alimentent la cellule ; large = sources en désaccord). Pour repasser en dollars, multiplier par un coût-tâche de référence (p.ex. agentique ≈ $1,99/tâche pour Opus 4.8 @max sur l'AA Index).

4 · Assigner les défauts — ratios par couple

Pour chaque couple de modèles A/B, chaque source qui les mesure au même effort donne un point = ratio A/B. Couleur = niveau d'effort (les clusters) ; gris = run @default (effort à assigner). Regarde près de quel cluster tombe chaque point gris et dis-moi son effort. Échelle log ; survole un point pour la source.

Chaque couple a deux axes empilés — coût (haut) et tokens (bas) — chacun à sa propre échelle log. Un trait pointillé relie le point coût et le point tokens quand ils viennent de la même source. Peu de sources publient des tokens → l'axe tokens est souvent vide. Zoom §4 : Shift+molette sur un axe zoome son échelle (les axes/points se redessinent), glisser = déplacer, double-clic = réinitialiser.

5 · Ce qui déplace tout — les critères, avec exemples

La matrice donne un ordre de grandeur. Ces quatre facteurs, chacun mesuré, la modulent fortement. Chaque carte donne un exemple chiffré réel.

Type de tâche

L'effort et le choix du modèle ne pèsent pas pareil selon la nature du travail.

Code review — Haiku 4.5 obtient un F1 de 0,365 à 1/3 du coût de Sonnet 4.6 (0,343). « Bigger isn't better » (arxiv 2606.15689).
Raisonnement — Opus 4.5 medium→high : +29 % de score pour 2,8× tokens (PostTrainBench). Là, l'effort paie.
Coding pur — relever le budget d'effort : +0,27 pt sur SWE-Bench Pro (2606.17930). Plateau.

Complexité

Plus la tâche est dure, plus l'effort coûte — jusqu'au point où il nuit.

192 runs stratifiés (whitekumalabo) : tâche facile ≈ $0,023 quel que soit l'effort ; tâche dure (L3) low→max = ×1,9.
U inversé — Sonnet 4.6 sur orchestration : low 0 % / medium 16,7 % / high 6,7 % — high dégrade (2606.08878).

Génération

Les Opus récents font plus avec moins ; Sonnet 5 est l'exception verbeuse.

Opus 4.8 vs 4.7, même migration DB : 4.8 utilise ~moitié des tokens, −40 % de coût (FullStack, 5 runs).
Sonnet 5 génère ~2,5× les tokens d'Opus 4.8 sur l'AA Index → $2,29/tâche > $1,99 malgré un tarif/token plus bas.

Cache & harness

Deux facteurs invisibles qui peuvent dominer le coût et le score.

Cache — même changement de code, cache cassé sur Sonnet : $10,30 vs $3,83 pour Opus (GitHub copilot #1770). L'agentique tourne à 90–96 % de cache-read.
Harness — Opus 4.8 sur Terminal-Bench : 84,6 % (AA) vs 71,9 % (Vals), même modèle. Confound n°1.

6 · Le régime sans thinking — le moins cher

Certaines sources tournent sans pensée étendue (reasoning off / API brut). Ce n'est pas un point du balayage low→max : c'est un régime de coût distinct, le plancher. On le traite couple-atomiquement, exactement comme la §2 : chaque source qui mesure deux modèles au même régime sur la même tâche donne un ratio de coût ; on médiane par couple. L'indice coût par modèle (Haiku 4.5 = 1×) est chaîné le long des ratios mesurés — indicatif, pas une décomposition. Fable 5 et Sonnet 5 n'ont aucun run no-think publié → absents ici. Haiku 4.5 vit presque entièrement dans ce régime (il n'expose pas l'effort discret).

7 · Confiance & méthode de fusion

Ce qui est solide

  • HAUTE Ancrages cross-modèle @medium (÷Opus 4.8) : Fable ~2,1×, Sonnet 4.6 ~0,7×, Haiku 4.5 ~0,19× (régime no-think, §6) — plusieurs méthodes indépendantes concordantes.
  • MOYENNE Pente d'effort en coût low→max : ×1,9 (Sonnet 4.6) à ×4–5 (Opus 4.7 / Sonnet 5) — plus raide sur le harnais agentique (CursorBench) que sur computer-use (OSWorld).
  • MOYENNE Plateau qualité + U inversé sur code/agentique (multi-papiers).

Ce qui reste large

  • LARGE Sonnet 5 : médiane mesurée ~0,7× Opus 4.8 (moins cher), mais queue verbeuse jusqu'à ~1,6× sur long agentique (AA) — d'où l'IC large.
  • LARGE Opus 4.7 vs 4.8 : sources contradictoires (effort/harness) → 0,6–1,4×.
  • LARGE Queues xhigh/max et modèles sans sweep d'effort mesuré.

Comment le coût relatif est fusionné — le procédé

  1. Ratios à tâche identique. On ne compare jamais deux dollars bruts de sources différentes (tailles de tâche incomparables). On ne retient que les sources mesurant ≥2 modèles/efforts sur la même tâche : leur rapport annule la variance de tâche. Ex. : WorkBench donne Sonnet 4.6/Opus 4.8 = 0,58 sur ses tâches, quel que soit leur coût absolu.
  2. Le couple (modèle, effort) est atomique. Chaque nœud = un couple indivisible. On ne suppose jamais que « l'effort » ait le même effet d'un modèle à l'autre : passer Haiku à max n'est pas le même multiplicateur que passer Opus à max. Pas de décomposition modèle×effort — la séparabilité serait une hypothèse fausse.
  3. Graphe de liaison. Chaque arête = un ratio mesuré entre deux couples sur la même tâche (deux modèles au même effort, ou deux efforts du même modèle). On place un couple en multipliant les ratios le long du chemin qui le relie à la base Opus 4.8 @medium. Plusieurs chemins → sa valeur = la médiane des produits (robuste aux confounds cache/harness). Ex. : Fable/Opus 4.8 à config appariée vu à 1,95 (WorkBench), 2,0 (ctala), 1,68 (X-post) → 1,95.
  4. « Default » ≠ un effort unique — vérifié source par source. On a lu la config réelle de chaque source (colonne Config ✓). Résultat : le bucket « défaut » est hétérogène — certaines à max (CEO-Bench, EMB-Opus), high (SlopCode, OfficeQA), pensée désactivée (STAGE-Claw, ianlpaterson), d'autres non précisées. La doc Anthropic explique pourquoi : le défaut dépend du modèle (Sonnet 5/Fable 5 pensent toujours ; Opus 4.x & Sonnet 4.6 sur opt-in ; Haiku & ≤4.5 sans le paramètre). Ce qui autorise un ratio, c'est le fait vérifié que chaque source applique la même config à tous ses modèles — pas un effort supposé.
  5. Couple non relié = signalé, jamais inventé. Un couple sans arête mesurée (Haiku @max — effort non exposé ; ou une sweep d'effort jamais publiée pour ce modèle) n'est pas rempli par une règle d'effort générique. La forme d'effort d'un modèle ne sert jamais de gabarit pour un autre : soit n/a, soit estimé avec une bande explicitement large.
  6. Confiance = concordance. L'intervalle vient de la dispersion des chemins concurrents. 3+ sources serrées → bande étroite (Sonnet 4.6 & Opus 4.8 bas effort). Sources en désaccord → bande large (Fable max : Willison-SVG vs CursorBench ; Sonnet 5, dépendant de la taille de tâche). Là où rien ne confirme, la bande le dit.
